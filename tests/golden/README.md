# 골든 픽스처 — 윤문 품질 회귀 게이트

taxonomy·플레이북·프롬프트를 고칠 때 **윤문 품질이 후퇴하지 않았는지**를 LLM 없이,
순수 Python으로 검증하는 회귀 방어선입니다. 모든 픽스처는 실사용에서 실제로 터진
실패 모드를 재현하는 **합성 텍스트**이며, 서지사항·인명·수치는 전부 허구입니다.

## 구조

```
tests/golden/
├── checks.py                  # 결정적 채점기 (stdlib only)
├── fixtures/
│   └── <NN_실패모드-이름>/
│       ├── input.txt              # 윤문 전 원문
│       ├── bad_output.txt         # 실패 사례 재현 (채점기가 반드시 FAIL시켜야 함)
│       ├── good_output.txt        # 기대 수준의 정상 윤문 (반드시 PASS해야 함)
│       └── expected_failures.json # bad가 트리거해야 할 실패 코드 목록
└── README.md
tests/test_golden.py           # pytest / unittest 겸용
```

## 실행

```bash
# 전체 (pytest 또는 unittest)
pytest tests/test_golden.py
python3 -m unittest tests.test_golden

# 채점기 단독 — 실제 윤문 결과를 게이트에 통과시킬 때
python3 tests/golden/checks.py fixtures/01_register_downgrade/input.txt <윤문결과.txt>
```

CI에서는 LLM을 부를 수 없으므로 자동 테스트는 **채점기 자체**를 양방향으로 검증합니다
(bad → FAIL, good → PASS, 무변경 → PASS). 실제 파이프라인을 돌린 뒤에는 위 CLI로
윤문본을 직접 채점하세요.

## 설계 원칙 — 방향성 게이트

LLM 출력은 비결정적이라 "정확히 이 문자열"을 기대하면 안 됩니다. 모든 체크는
**"이 카운트가 늘면/이 요소가 사라지면 실패"** 방향 조건입니다. 예:

- `하였` 이 원문보다 **늘면** FAIL (원문에 이미 있으면 유지·감소 허용)
- 상투구("기록적인 성과", "~로 평가된다" 등)가 **새로 생기면** FAIL
- 구어 종결(~요)이 원문에 3회 이상인데 절반 미만으로 **격감하면** FAIL
- 제목 줄이 독립 줄에서 사라지면 FAIL (본문 흡수/완전 소실 구분 보고)
- 각주 개수 감소·번호 집합 변화·인접 수치 앵커 이탈·정의 변조 시 FAIL
- 「」·“”·따옴표 직접 인용(8자 이상)이 원문 그대로 없으면 FAIL

오탐을 극도로 경계합니다 — 정상 윤문을 FAIL시키는 게이트는 아무도 안 씁니다.
확신할 수 없는 조건은 넣지 않았고, 못 잡는 것은 아래에 명시합니다.

## 못 잡는 것 (정직한 한계)

- 의미 드리프트(없던 주장 주입), 과잉 은유, 어색한 문학체 — regex로 판정 불가
- 따옴표·수치가 없는 문장에 붙은 각주의 **위치** 이동 (개수·번호는 잡음)
- 리라이터가 제목을 문구만 다듬은 경우 — 소실로 간주됨 (제목은 원문 그대로가 철칙)
- 각주 표기는 한국 학술 관례(`…했다.1)` 처럼 붙여 씀)를 전제. 띄어 쓴 `1)` 은
  각주 정의 라벨로 해석됩니다.

## 픽스처 추가하는 법

1. `fixtures/NN_실패모드-이름/` 디렉토리를 만듭니다.
2. `input.txt` — 실패 모드를 **선명하게** 유발하는 한국어 원문. 실사용 텍스트를
   그대로 넣지 마세요(공개 레포입니다). 같은 실패를 재현하는 합성 텍스트를 쓰고,
   인명·서지·수치는 허구로 지어냅니다.
3. `bad_output.txt` — 그 실패가 실제로 일어난 모습. 애매하게 만들지 마세요.
4. `good_output.txt` — 철칙(내용·인용·구조·register 불변)을 지킨 정상 윤문.
5. `expected_failures.json` — `{"description": "...", "bad_must_fail": [코드들]}`.
   코드 목록은 `checks.py` 상단 docstring 참조.
6. `python3 -m unittest tests.test_golden` 이 통과하는지 확인합니다. 새 실패
   모드가 기존 체크로 안 잡히면 `checks.py`에 방향성 체크를 추가하되, 반드시
   무변경(identity)·정상 윤문이 PASS함을 함께 증명하세요.

## 원천

같은 엔진의 상용 웹앱(imnotai) 골든 회귀셋 체계를 이식했습니다. 픽스처 01의
입력은 웹앱 골든셋의 합성 샘플(실사용 ★1 평가에서 관찰된 격식 역주행 실패
모드를 재현하도록 제작된 것)이고, 픽스처 02는 실사용 학술논문에서 발생한
구조·각주 파괴 사고를 재현한 신규 합성입니다.
